How This Was Built
The pipeline architecture and key design decisions behind this assessment.
D1 Evidence Over Narrative

Recruiter-facing fit pages typically produce AI-generated prose verdicts like "Brian is a great fit because…" — which can't be verified and erodes trust the moment it oversells.

This pipeline shows evidence instead: inspectable code links, session counts, and match strength bars. The grade follows from the evidence; it doesn't explain it away.

D2 Three-Tier Evidence Model

Skills are color-coded by how verifiable they are: Inspectable (green) — the code is on GitHub. Deployed (blue) — there's a live URL. Claimed (gray) — resume only, no external proof.

This lets a hiring manager make informed trust decisions without reading every line. Most resume skills are gray. The interesting signal is green.

D3 Taxonomy-Driven Matching

Job requirements don't use canonical skill names. "Experience with LLMs" and "hands-on ML work" are different phrasings of the same signal. The matcher uses a 104-entry taxonomy with aliases and fuzzy resolution to normalize both sides before scoring.

This eliminates the false negatives that naive string matching produces.

D4 Grade Calibration Against Real Postings

Grades are not relative to the candidate's best self — they're calibrated against {{ .Params.benchmark_postings | default 61 }} real job postings with known expected outcomes. An A means the signal profile matches what A candidates actually look like for this role type.

The benchmark suite runs on every pipeline change to detect score drift.
